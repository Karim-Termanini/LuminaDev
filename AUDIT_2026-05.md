# LuminaDev — Full Codebase Audit Report

**Date:** 2026-05-26 | **Branch:** `feat/dashboard-fixes` | **Auditor:** Claude Sonnet 4.6

---

## 1. Executive Summary

LuminaDev is a well-structured, genuinely ambitious Tauri+React developer dashboard. The Tauri migration is functionally complete, the core Docker/Git/SSH/Monitor surfaces work end-to-end, and a credible CI gate exists. However, this audit identifies **twenty-seven distinct gaps** across five categories: architectural violations, phantom settings contracts, static/mock data still present in production paths, incomplete phase deliverables that are incorrectly marked done, and stale or contradictory documentation. None of these gaps break the primary user flows, but several create **silent lies** to the user — UI controls that appear functional but persist to a store that no backend code ever reads.

---

## 2. Architectural Integrity

### 2.1 `lib.rs` Monolith Violation — CRITICAL

**Finding:** `lib.rs` is **5,026 lines** containing **48 functions**. The architecture standard in both `CLAUDE.md` and `phasesPlan.md` mandates this file be a "thin dispatcher < 300 lines" with one line per handler. The 6-module refactoring plan (Phase 16 follow-up) identifies the correct split into `utils.rs`, `docker_ext.rs`, `terminal_pty.rs`, `ssh_ext.rs`, `git_parser.rs`, `runtime_installer.rs` — but this has not occurred.

**Impact:** `docker_install_invoke` (L843–942), `docker_remap_port_invoke` (L1202–1390), and `dh:profile:switch` (L2046–2165) each contain 80–140+ lines of inline business logic, directly violating the "> 50 lines in lib.rs → extract to domain module" red flag. Test isolation is compromised because these handlers cannot be unit-tested without the full Tauri runtime.

**Partial remediation exists:** `compose_profiles.rs`, `runtime_jobs.rs`, `git_vcs_network.rs`, `git_vcs_file_diff.rs`, `git_vcs_repo_state.rs`, `git_vcs_ipc.rs`, `cloud_git_ipc.rs`, `cloud_auth.rs`, `readiness.rs`, `readiness_ipc.rs`, `project_scaffold.rs`, `profile_credentials.rs`, and various runtime modules have been extracted. However, the largest domain (Docker) and the profile switch engine remain inline. The monolith is **17× over the stated limit**.

### 2.2 `removableDeps` Hardcoded Empty Array

**Finding:** At `lib.rs:3785`:

```rust
"removableDeps": [],
```

The `runtime:uninstall:preview` response always returns an empty `removableDeps` array. Acknowledged in `phasesPlan.md` ("real dep graph — `removableDeps` always empty") but not communicated to users.

**Impact:** The "Remove with dependencies" option in `RuntimesPage.tsx:1085` renders a conditionally visible section that will **never appear** because `removableDeps` is always empty. The UI control exists but is permanently invisible. Users cannot know whether removing a runtime will orphan shared packages.

## 3. Phantom Settings Contracts — HIGH SEVERITY

This is the most pervasive category of issues. The Settings page (`/settings`) has eleven sub-pages that persist values to the Tauri store — but the following stored keys are never read back by any Rust handler to alter runtime behavior. They are **write-only configuration stores**.

### 3.1 `resources_settings` — CPU/RAM Limits Not Enforced

`SettingsResources.tsx` presents sliders for `cpuMaxPct` and `ramMaxMb`. These write to `resources_settings` in the store. A search of all `.rs` files for `resources_settings`, `cpu_limit`, `cpuMaxPct`, and `ramMaxMb` returns **zero matches**. The Rust job runner spawns `tokio::spawn` tasks with no CPU or memory constraints regardless of what is saved. The `phasesPlan.md` Phase 8 scope says "sliders and inputs for defining strict CPU limits and RAM allocations **that the engine respects during job execution**" — the second half is entirely absent.

### 3.2 `app_engine_settings` — IPC Timeouts Not Wired

`SettingsAppEngine.tsx` stores `ipcTimeoutMs` (range 1000–120000ms), `threadPoolSize`, and `daemonAutoRestart`. The Rust backend uses hardcoded constants: `CMD_TIMEOUT_SHORT = 30s`, `CMD_TIMEOUT_DEFAULT = 180s`, `CMD_TIMEOUT_LONG = 900s`. These constants are compile-time fixed in `lib.rs`. The stored `ipcTimeoutMs` is never read. A user who sets "timeout = 5 seconds" to reduce hang risk gets no protection.

### 3.3 `update_settings` — Auto-Update Is UI Theater

`SettingsUpdate.tsx` saves `releaseChannel` ("stable"/"alpha"), `checkOnStartup`, and `lastChecked`. No Tauri updater plugin is configured. No `dh:app:update:check` IPC channel exists in `lib.rs`. `checkOnStartup` is saved but never read at app launch. `lastChecked` will remain `null` forever since no check ever fires. The "Last checked" display will always show "Never checked."

### 3.4 `notification_settings` — No Notification Dispatch System

`SettingsNotification.tsx` saves `globalMute`, `minSeverity` ("info"/"warn"/"error"), and `osNotifications` (hardcoded disabled: "Requires Tauri notification plugin (Phase 10)"). The entire app's notification layer uses React `useState` toasts that are never filtered through this store. Nothing reads `notification_settings` before showing a toast. `globalMute: true` will not silence anything.

### 3.5 `shortcuts_settings` — Key Capture Disconnected from Dispatch

`SettingsShortcuts.tsx` correctly captures key chords and saves them to `shortcuts_settings`. However, a search for `shortcuts_settings` consumers across all `.tsx` and `.ts` files (excluding the settings page itself) returns **zero results**. No global `keydown` listener in `AppShell.tsx` or `App.tsx` reads these bindings. Pressing the custom chord for `go_docker` navigates nowhere. The six defined actions (`open_terminal`, `toggle_sidebar`, `focus_search`, `go_dashboard`, `go_docker`, `go_git`) are permanently disconnected from the router.

### 3.6 `language_settings` — i18n Is Fake

`SettingsLanguages.tsx` lists five future locales (fr-FR, de-DE, es-ES, ar-SA, zh-CN) as `disabled` with "coming soon" labels. The save button writes `{ locale: 'en-US' }` unconditionally and calls `document.documentElement.lang = 'en'`. No i18n library (react-intl, i18next, @formatjs, or similar) is present anywhere in the codebase. The dropdown is permanently locked to English. This is documented inline ("Additional languages coming in a future release") but the presence of a functional-looking UI with a Save button implies capability that doesn't exist.

### 3.7 `beta_features_state` — Feature Flags Consumed Nowhere

`SettingsBetaFeatures.tsx` allows toggling experimental flags. A search for `beta_features_state` consumers returns zero results outside the settings page. No code path reads these flags to gate feature visibility.

## 4. Static and Mock Data in Production Paths

### 4.1 Dashboard Profile Cards — Four Permanently Disabled Profiles

`DashboardMainPage.tsx:51–54` defines four preset profile cards with `status: 'planned'`:

- Mobile App Dev — "React Native / Flutter environment stub."
- Game Development — "Godot/Unity/Unreal minimal engine stub."
- Infra / K8s — "Local minikube/k3d or Terraform runner stub."
- Desktop Qt/GTK — "Native desktop application build environment."

These cards render in the UI but the "Set Active" button is `disabled` and the opacity is reduced. `phasesPlan.md` claims Phase 9 is ✅ DONE with mobile and AI/ML scaffolds fully implemented — but the dashboard surface still shows them as `planned`. The scaffold implementation exists in `project_scaffold.rs` and `DashboardMainPage.tsx` has `dh:project:scaffold` IPC calls for mobile and AI/ML, but the profile card grid still serves static `status: 'planned'` labels for those profiles.

### 4.2 `custom.placeholder` Widget Still Registered

`DashboardWidgetDeck.tsx:195` has an active `case 'custom.placeholder'` that renders "Placeholder for Phase 1 custom profiles and user-defined widget packs." This was Phase 1 scaffolding. If any user's `dashboard-layout.json` contains a `custom.placeholder` widget instance, it will render this text in production.

### 4.3 AppStream `metainfo.xml` — Three Factual Errors

`data/io.github.karimodora.LinuxDevHome.metainfo.xml` contains:

1. **Wrong version:** `<release version="0.1.0" date="2026-04-28">` — the project is at `v0.2.0-alpha`, tagged and released. This is the version Flathub will display.
2. **Wrong homepage URL:** `https://github.com/Karim-Termanini/PersonalProjekte` — this is a personal projects repository, not the LuminaDev repository. Clicking the homepage from Flathub or GNOME Software would navigate to the wrong repo.
3. **Incorrect technology description:** "Initial public skeleton: Electron UI" — the Electron migration was completed. Flathub reviewers checking this description against a Tauri app will find a contradiction.
4. **No `<screenshots>` section** — Flathub submission guidelines require at least one screenshot for the app to pass review. The checklist item "AppStream Metadata: metainfo.xml with license, summary, and screenshots" in Phase 14 is therefore not complete.

### 4.4 GPU Detection Fallback Is Misleading Static Data

`DashboardKernelsPage.tsx:56–59`:

```typescript
setGpu(g.ok && typeof g.result === 'string' ? g.result : 'Intel Integrated Graphics')
```

When `nvidia-smi` fails (any non-NVIDIA system, any Flatpak session), the displayed GPU is hardcoded to `'Intel Integrated Graphics'`. A user with an AMD GPU, a discrete Radeon card, or a system with no dedicated GPU at all will see "Intel Integrated Graphics." This is fabricated data presented as a detected hardware fact.

## 7. Security Surface Observations

### 7.1 Docker Install Password Transmitted in IPC Body

`docker_install_invoke` at `lib.rs:843` accepts `body.get("password")` — the sudo password is transmitted as a plaintext JSON field over the Tauri IPC bridge. While Tauri IPC is process-local (not a network socket), the password is stored as a `serde_json::Value` string in Rust memory during the install operation. The `phasesPlan.md` Phase 16 spec recommends `pkexec` (Polkit) for privilege escalation — that approach does not require a password in the payload at all.

### 7.2 `sshpass` Password in Environment Variable

The SSH remote key setup uses `sshpass`, which passes the password through an environment variable or process arguments. Environment variables are visible in `/proc/self/environ` to processes running as the same user, and may appear in process listings.

### 7.3 No Input Sanitization Acknowledgment for `distro` Field

The `distro` field from the request body is correctly validated against a `matches!` allowlist. However, the error message at `lib.rs:880` echoes the raw `host_distro_id` string read from `/etc/os-release` back into the IPC response. This is acceptable in the current local-read context but should be noted if the path ever processes remote input.

### 8.1 OS Notifications Disabled and Gated Behind Phase 10

`SettingsNotification.tsx` renders the "OS native notifications" toggle as `checked={false}` with `onChange={() => {}}` and `disabled` — permanently off. The note says "Requires Tauri notification plugin (Phase 10)." Phase 10 is post-Alpha and not started. ### 8.3 Per-Container Stats Stream Absent

`phasesPlan.md` Phase 5 lists per-container stats stream as missing. `DashboardKernelsPage.tsx` does not call `docker stats` or any streaming Docker stats API. The monitor page shows Docker container count but no per-container CPU/memory breakdown.

`SettingsUpdate.tsx` displays a "Check for updates" control with a `checkOnStartup` toggle. `App.tsx` startup code does not call any update check at launch. No update-related channel exists in `lib.rs`. The `tauri-plugin-updater` is not present. The entire update subsystem is future work presented as a configurable setting.
