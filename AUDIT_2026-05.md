## 4. CPU Resource Limit Enforcement — ❌ RESOLVED (tab removed)

**Status:** The `SettingsResources` tab has been removed entirely.

**Rationale:** The `cpuLimitPercent` slider saved to store but had zero runtime effect — no Rust handler reads it. CPU percentage limiting on Linux requires cgroups v2, `cpulimit`, or `systemd-run`, none of which are appropriate for this project. The `ramLimitMb` slider only applied to Java/Go via env vars, not as a general memory governor. The only meaningful resource control — `threadPoolSize` (concurrent job semaphore) — already lives in Settings → App Engine where it belongs.

**What remains:** `executor.rs` uses hardcoded defaults (80% core count, 4GB RAM) for env-var enforcement on runtime install steps (`CARGO_BUILD_JOBS`, `GOMEMLIMIT`, `_JAVA_OPTIONS`, `NODE_OPTIONS`). These are not user-configurable — they're sensible defaults for all runtime install/update/remove operations.

**Files changed:**

- Deleted `apps/desktop/src/renderer/src/pages/settings/SettingsResources.tsx`
- Removed import, nav entry, subtype keys from `SettingsShell.tsx`
- Removed `ResourcesSettingsSchema`, type, store key from `packages/shared/src/schemas.ts`
- Removed `"resources_settings"` from Rust store allowlist (`utils.rs`, `lib.rs`)
- Removed 10 i18n keys from all 3 locales (en-US, de-DE, ar-SA)
- `executor.rs` switched from store lookup to hardcoded const defaults

## 10. Security Surface

### 10.1 Profiles Page — Container Status Detection + Multi-Profile Support — ❌ OPEN (redesign required)

**Checked:** `ProfilesPage.tsx` — multiple fix attempts have failed. Root cause identified.

**Fix attempts (all insufficient):**

1. `dh:host:exec` with `maintenance_docker_ps_table` → output truncation + case sensitivity issues
2. `window.dh.dockerList()` + container name prefix matching → fragile; broken by profile names with special chars
3. `dh:profile:running-status` backend endpoint → same prefix-matching bug
4. `compose_engine.rs`: switched from `COMPOSE_PROJECT_NAME` env var to `-p` CLI flag — correct but insufficient alone
5. `system_info.rs`: fixed name sanitization in `handle_profile_running_status` — correct but still prefix-matching

**Root cause (definitive):** `docker/compose/*/docker-compose.yml` files all have hardcoded `name: lumina-*` fields (e.g., `name: lumina-web-dev`, `name: lumina-data-science`). In some Docker Compose v2 versions the `name:` YAML field overrides even the `-p` CLI flag. Containers launched before the fix already exist under `lumina-*` project names. Name-prefix matching against `docker ps` output is fundamentally fragile and will keep breaking.

**Required fix — `docker compose ls` as the source of truth:**

`docker compose ls --format json` returns the authoritative list of projects Docker actually tracks, by exact project name:
```json
[{"Name":"testeing","Status":"running(2)","ConfigFiles":"..."},...]
```
Backend `handle_profile_running_status` should call this instead of parsing `docker ps`. Frontend `checkRunning` compares exact sanitized profile names against the `Name` field. No guessing, no substring matching.

**Required architectural change — multi-profile support:**

User requirement: multiple profiles can run simultaneously. Current `profile_switch` (line 165–207 of `profile_engine.rs`) stops ALL other compose projects before starting the target profile. This must be removed.

New model:
- **Start button**: calls `dh:compose:up` directly (not `dh:profile:switch`). Does NOT stop others.
- **Stop button**: calls `dh:compose:down` for that profile only.
- **Restart button**: calls `dh:compose:down` then `dh:compose:up` for that profile only.
- **"Set Active" concept**: the `active_profile` store key tracks which profile is "primary" for Dashboard widgets — separate from running/stopped state.
- **Empty state**: if no profiles are running, show neutral "nothing running" state rather than showing a running indicator.

**Files to change:**
- `apps/desktop/src-tauri/src/system_info.rs` — `handle_profile_running_status`: use `docker compose ls --format json` + exact name match
- `apps/desktop/src-tauri/src/profile_engine.rs` — `profile_switch`: remove the "pause other profiles" block (lines 165–207)
- `apps/desktop/src/renderer/src/pages/ProfilesPage.tsx` — Start button: call `dh:compose:up` instead of `dh:profile:switch`; `checkRunning`: use `dh:profile:running-status` once backend is fixed
- `docker/compose/*/docker-compose.yml` — remove all `name: lumina-*` fields so `-p` flag is the only project name source

### 10.2 sshpass Password Visibility — ✅ FIXED

**Prior audit incorrectly reported `-p` flag** — this was already switched to `-e` before the audit. ✅ **Fixed.**

### 10.3 Runtime Install Sudo Password — ✅ FIXED

**Checked:** `runtime_jobs.rs`, `DockerPage.tsx`, and `RuntimesPage.tsx` no longer handle or transmit a `sudo_password` field via IPC. The application now correctly relies entirely on the system's native Polkit (`pkexec`) dialog for privilege escalation during both Runtime and Docker installations, ensuring credentials never traverse the Tauri IPC boundary.

**Verdict:** ✅ FIXED. Sudo password state completely removed.

### 10.4 `/tmp` Predictable Path — ✅ FIXED

**Checked:** `system_info.rs` uses the `tempfile::NamedTempFile` crate to securely create temporary files for modifying `/etc/hosts`, avoiding TOCTOU and symlink attacks associated with predictable `/tmp` paths.

**Verdict:** ✅ FIXED.

**Remaining `/tmp/lumina-*` paths in `runtime_jobs.rs` (shell script strings for Go/Zig/Java downloads):** These are temporary download files inside bash scripts that are immediately `rm -f`'d after extraction. No TOCTOU risk in bash scripts because download + extract + delete is a single sequential step. Acceptable pattern.

### 10.5 Prior P0: Docker Install Password in IPC Body — ✅ FIXED

Confirmed: `docker_engine.rs` contains no `password` field handling. Readiness wizard uses `pkexec` (polkit native dialog) at `readiness.rs:232`. This concern is resolved. ✅
