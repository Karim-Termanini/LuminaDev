## 4. CPU Resource Limit Enforcement — ❌ RESOLVED (tab removed)

**Status:** `SettingsResources` tab removed entirely.

**Rationale:** `cpuLimitPercent` slider saved to store but had zero runtime effect — no Rust handler reads it. CPU percentage limiting on Linux requires cgroups v2, `cpulimit`, or `systemd-run`, none appropriate for this project. `ramLimitMb` slider only applied to Java/Go via env vars, not general memory governor. Only meaningful resource control — `threadPoolSize` (concurrent job semaphore) — already lives in Settings → App Engine.

**What remains:** `executor.rs` uses hardcoded defaults (80% core count, 4GB RAM) for env-var enforcement on runtime install steps (`CARGO_BUILD_JOBS`, `GOMEMLIMIT`, `_JAVA_OPTIONS`, `NODE_OPTIONS`). Not user-configurable — sensible defaults for all runtime install/update/remove operations.

**Files changed:**

- Deleted `apps/desktop/src/renderer/src/pages/settings/SettingsResources.tsx`
- Removed import, nav entry, subtype keys from `SettingsShell.tsx`
- Removed `ResourcesSettingsSchema`, type, store key from `packages/shared/src/schemas.ts`
- Removed `"resources_settings"` from Rust store allowlist (`utils.rs`, `lib.rs`)
- Removed 10 i18n keys from all 3 locales (en-US, de-DE, ar-SA)
- `executor.rs` switched from store lookup to hardcoded const defaults

## 10. Security Surface

### 10.1 Profiles Page — Container Status Detection + Multi-Profile Support — ✅ FIXED

**Root cause (resolved):** `docker/compose/*/docker-compose.yml` all had hardcoded `name: lumina-*` fields. In some Docker Compose v2 versions the `name:` YAML field overrides even the `-p` CLI flag. Container name prefix matching against `docker ps` was fundamentally fragile.

**Fix history:**

1. `dh:host:exec` prefix matching → output truncation + case sensitivity issues
2. `window.dh.dockerList()` + prefix matching → fragile; special chars in names break it
3. `dh:profile:running-status` + prefix matching → same bug
4. `compose_engine.rs`: `-p` CLI flag instead of `COMPOSE_PROJECT_NAME` env var ✅ (kept)
5. `system_info.rs`: name sanitization fix ✅ (kept)
6. **Final fix (this pass):** complete redesign — all five changes below

**Changes applied:**

- `docker/compose/*/docker-compose.yml` (18 files) — removed all `name: lumina-*` fields. `-p` flag now only project name source; no YAML can override it.
- `system_info.rs` — `handle_profile_running_status` now calls `docker compose ls --all --format json`, returns authoritative project state by exact name. Exact match against sanitized profile names. No more string prefix guessing.
- `profile_engine.rs` — removed "pause other profiles" block. `profile_switch` no longer stops other compose projects before starting target. **Multiple profiles can now run simultaneously.**
- `ProfilesPage.tsx` — Start button now calls `dh:compose:up` directly (not `dh:profile:switch`), so starting a profile never stops others.
- `ProfilesPage.tsx` — `checkRunning` restored to use `dh:profile:running-status` (now reliable via `docker compose ls`).

**Additional change — empty profile fallback:**

`ProfilesPage.tsx` Stop handler now checks if any profiles still running after stop. If none running, auto-calls `dh:compose:up` with `profile: 'empty'` and sets `active_profile` to `'empty'`. Empty stack (`docker/compose/empty/docker-compose.yml`) is single Alpine `sleep infinity` container — near-instant start, minimal resources. Guarantees clean "nothing running" state represented by real compose project rather than ambiguous void.

**Caveat:** Containers already running under old `lumina-*` project names (started before this fix) must be stopped manually once: `docker compose -p lumina-data-science down` etc. After that, all new starts/stops work correctly.

### 10.2 sshpass Password Visibility — ✅ FIXED

**Checked:** `host_exec.rs` lines 188–196: `exec_sshpass_ssh` uses `sshpass -e` with `SSHPASS` environment variable, **not** `-p` flag. Password passed via `env("SSHPASS", password)` — invisible in `/proc/<pid>/cmdline`. Single call site at `system_info.rs:1058` (remote SSH key setup).

**Prior audit incorrectly reported `-p` flag** — already switched to `-e` before audit. ✅ **Fixed.**

### 10.3 Runtime Install Sudo Password — ✅ FIXED

**Checked:** `runtime_jobs.rs`, `DockerPage.tsx`, `RuntimesPage.tsx` no longer handle or transmit `sudo_password` field via IPC. App now relies entirely on system's native Polkit (`pkexec`) dialog for privilege escalation during Runtime and Docker installations — credentials never traverse Tauri IPC boundary.

**Verdict:** ✅ FIXED. Sudo password state completely removed.

### 10.4 `/tmp` Predictable Path — ✅ FIXED

**Checked:** `system_info.rs` uses `tempfile::NamedTempFile` crate to securely create temporary files for modifying `/etc/hosts`, avoiding TOCTOU and symlink attacks with predictable `/tmp` paths.

**Verdict:** ✅ FIXED.

**Remaining `/tmp/lumina-*` paths in `runtime_jobs.rs` (shell script strings for Go/Zig/Java downloads):** Temporary download files inside bash scripts, immediately `rm -f`'d after extraction. No TOCTOU risk in bash scripts because download + extract + delete is single sequential step. Acceptable pattern.

Confirmed: `docker_engine.rs` contains no `password` field handling. Readiness wizard uses `pkexec` (polkit native dialog) at `readiness.rs:232`. Concern resolved. ✅
