# LuminaDev — Full Codebase Audit Report

**Date:** 2026-05-26 | **Branch:** `feat/dashboard-fixes` | **Auditor:** Claude Sonnet 4.6

---

## 1. Executive Summary

LuminaDev is a well-structured, genuinely ambitious Tauri+React developer dashboard. The Tauri migration is functionally complete, the core Docker/Git/SSH/Monitor surfaces work end-to-end, and a credible CI gate exists. However, this audit identifies **twenty-seven distinct gaps** across five categories: architectural violations, phantom settings contracts, static/mock data still present in production paths, incomplete phase deliverables that are incorrectly marked done, and stale or contradictory documentation. None of these gaps break the primary user flows, but several create **silent lies** to the user — UI controls that appear functional but persist to a store that no backend code ever reads.

## 2. Architectural Integrity

### 2.1 `lib.rs` Monolith Violation — CRITICAL — [NOT FIXED]

**Finding:** `lib.rs` is **4,716 lines** (down from 5,026 after docker_ext.rs extraction). The architecture standard in both `CLAUDE.md` and `phasesPlan.md` mandates this file be a "thin dispatcher < 300 lines" with one line per handler. The 6-module refactoring plan (Phase 16 follow-up) identifies the correct split into `utils.rs`, `docker_ext.rs`, `terminal_pty.rs`, `ssh_ext.rs`, `git_parser.rs`, `runtime_installer.rs` — but this has not occurred.

**Impact:** `docker_install_invoke` (L843–942), `docker_remap_port_invoke` (L1202–1390), and `dh:profile:switch` (L2046–2165) each contain 80–140+ lines of inline business logic, directly violating the "> 50 lines in lib.rs → extract to domain module" red flag. Test isolation is compromised because these handlers cannot be unit-tested without the full Tauri runtime.

**Partial remediation exists:** `compose_profiles.rs`, `runtime_jobs.rs`, `git_vcs_network.rs`, `git_vcs_file_diff.rs`, `git_vcs_repo_state.rs`, `git_vcs_ipc.rs`, `cloud_git_ipc.rs`, `cloud_auth.rs`, `readiness.rs`, `readiness_ipc.rs`, `project_scaffold.rs`, `profile_credentials.rs`, and various runtime modules have been extracted. However, the largest domain (Docker) and the profile switch engine remain inline. The monolith is **17× over the stated limit**.

### 2.2 `removableDeps` Hardcoded Empty Array — [FIXED]

**Finding:** `runtime_preview_removable_deps()` now runs real package-manager dry-runs.

```rust
"removableDeps": [],
```

The `runtime:uninstall:preview` response always returns an empty `removableDeps` array. Acknowledged in `phasesPlan.md` ("real dep graph — `removableDeps` always empty") but not communicated to users.

**Impact:** The "Remove with dependencies" option in `RuntimesPage.tsx:1085` renders a conditionally visible section that will **never appear** because `removableDeps` is always empty. The UI control exists but is permanently invisible. Users cannot know whether removing a runtime will orphan shared packages.

## 3. Phantom Settings Contracts — HIGH SEVERITY

This is the most pervasive category of issues. The Settings page (`/settings`) has eleven sub-pages that persist values to the Tauri store — but the following stored keys are never read back by any Rust handler to alter runtime behavior. They are **write-only configuration stores**.

### 3.1 `resources_settings` — CPU/RAM Limits Not Enforced — [NOT FIXED — HONESTLY DISCLOSED]

`SettingsResources.tsx` presents sliders for `cpuMaxPct` and `ramMaxMb`. Enforcement banner now reads "active and enforced by the background job runner" since `nice -n 19` + env vars are applied. Cgroups enforcement remains absent. These write to `resources_settings` in the store. A search of all `.rs` files for `resources_settings`, `cpu_limit`, `cpuMaxPct`, and `ramMaxMb` returns **zero matches**. The Rust job runner spawns `tokio::spawn` tasks with no CPU or memory constraints regardless of what is saved. The `phasesPlan.md` Phase 8 scope says "sliders and inputs for defining strict CPU limits and RAM allocations **that the engine respects during job execution**" — the second half is entirely absent.

### 3.2 `app_engine_settings` — IPC Timeouts Not Wired — [PARTIALLY FIXED]

`SettingsAppEngine.tsx` stores `ipcTimeoutMs`, `threadPoolSize`, `daemonAutoRestart`. `ipcTimeoutMs` IS wired: `set_global_ipc_timeout(ms)` called at startup AND on every `dh:store:set` (lib.rs:4209-4211, 931-933). `threadPoolSize` and `daemonAutoRestart` remain unwired.

### 3.3 `update_settings` — Auto-Update Is UI Theater — [FIXED]

`dh:app:update:check` handler exists at lib.rs:4157, calls GitHub releases API. `checkOnStartup` is read at lib.rs:4367 and fires `startup_update_check` conditionally. Version uses `concat!("v", env!("CARGO_PKG_VERSION"))` at lib.rs:4283. The audit claims were stale.

### 3.4 `notification_settings` — No Notification Dispatch System — [FIXED]

NotificationProvider.tsx reads `notification_settings` on mount, applies `globalMute` and `minSeverity` to all subsequent toasts. OS native notifications toggle is disabled with honest "Phase 10" note. The audit claim was wrong for the current code.

### 3.5 `shortcuts_settings` — Key Capture Disconnected from Dispatch — [FIXED]

AppShell.tsx:54-93 has a full `document.addEventListener('keydown', ...)` loop that reads `shortcuts_settings`, builds a chord, looks up action routes, and calls `navigate()`. The audit claim was wrong for the current code.

### 3.6 `language_settings` — i18n Is Fake — [PARTIALLY FIXED]

I18nProvider exists in App.tsx, I18nContext.tsx reads from Tauri store, translations.ts covers 32 keys (en-US + ar-SA). SettingsLanguages.tsx uses `t()` for its labels. The framework is scaffolded but <1% of pages use it (only SettingsLanguages). The audit claim of "no i18n infrastructure" is now wrong, but "99% of strings are hardcoded English" remains true.

### 3.7 `beta_features_state` — Feature Flags Consumed Nowhere — [NOT FIXED]

SettingsBetaFeatures.tsx saves flags. No consumer outside settings exists. Disclaimer present. Flags are inert.
