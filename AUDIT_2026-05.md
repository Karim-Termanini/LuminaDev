## 4. CPU Resource Limit Enforcement ÔÇö ÔŁî RESOLVED (tab removed)

**Status:** The `SettingsResources` tab has been removed entirely.

**Rationale:** The `cpuLimitPercent` slider saved to store but had zero runtime effect ÔÇö no Rust handler reads it. CPU percentage limiting on Linux requires cgroups v2, `cpulimit`, or `systemd-run`, none of which are appropriate for this project. The `ramLimitMb` slider only applied to Java/Go via env vars, not as a general memory governor. The only meaningful resource control ÔÇö `threadPoolSize` (concurrent job semaphore) ÔÇö already lives in Settings Ôćĺ App Engine where it belongs.

**What remains:** `executor.rs` uses hardcoded defaults (80% core count, 4GB RAM) for env-var enforcement on runtime install steps (`CARGO_BUILD_JOBS`, `GOMEMLIMIT`, `_JAVA_OPTIONS`, `NODE_OPTIONS`). These are not user-configurable ÔÇö they're sensible defaults for all runtime install/update/remove operations.

**Files changed:**

- Deleted `apps/desktop/src/renderer/src/pages/settings/SettingsResources.tsx`
- Removed import, nav entry, subtype keys from `SettingsShell.tsx`
- Removed `ResourcesSettingsSchema`, type, store key from `packages/shared/src/schemas.ts`
- Removed `"resources_settings"` from Rust store allowlist (`utils.rs`, `lib.rs`)
- Removed 10 i18n keys from all 3 locales (en-US, de-DE, ar-SA)

## 10. Security Surface

### 10.1 Profiles Page ÔÇö Container Status Detection + Multi-Profile Support ÔÇö Ôťů FIXED

**Root cause (resolved):** `docker/compose/*/docker-compose.yml` all had hardcoded `name: lumina-*` fields. In some Docker Compose v2 versions the `name:` YAML field overrides even the `-p` CLI flag. Container name prefix matching against `docker ps` was fundamentally fragile.

**Fix history:**

1. `dh:host:exec` prefix matching Ôćĺ output truncation + case sensitivity issues
2. `window.dh.dockerList()` + prefix matching Ôćĺ fragile; special chars in names break it
3. `dh:profile:running-status` + prefix matching Ôćĺ same bug
4. `compose_engine.rs`: `-p` CLI flag instead of `COMPOSE_PROJECT_NAME` env var Ôťů (kept)
5. `system_info.rs`: name sanitization fix Ôťů (kept)
6. **Final fix (this pass):** complete redesign ÔÇö all five changes below

**Changes applied:**

- `docker/compose/*/docker-compose.yml` (18 files) ÔÇö removed all `name: lumina-*` fields. `-p` flag is now the only project name source; no YAML can override it.
- `system_info.rs` ÔÇö `handle_profile_running_status` now calls `docker compose ls --all --format json`, which returns authoritative project state by exact name. Exact match against sanitized profile names. No more string prefix guessing.
- `profile_engine.rs` ÔÇö removed "pause other profiles" block. `profile_switch` no longer stops other compose projects before starting the target. **Multiple profiles can now run simultaneously.**
- `ProfilesPage.tsx` ÔÇö Start button now calls `dh:compose:up` directly (not `dh:profile:switch`), so starting a profile never stops others.
- `ProfilesPage.tsx` ÔÇö `checkRunning` restored to use `dh:profile:running-status` (now reliable via `docker compose ls`).

**Additional change ÔÇö empty profile fallback:**

`ProfilesPage.tsx` Stop handler now checks if any profiles are still running after a stop. If none are running, automatically calls `dh:compose:up` with `profile: 'empty'` and sets `active_profile` to `'empty'`. The empty stack (`docker/compose/empty/docker-compose.yml`) is a single Alpine `sleep infinity` container ÔÇö near-instant start, minimal resources. This guarantees a clean "nothing is running" state is always represented by a real compose project rather than an ambiguous void.

**Caveat:** Containers already running under old `lumina-*` project names (started before this fix) must be stopped manually once: `docker compose -p lumina-data-science down` etc. After that, all new starts/stops work correctly.
