## 4. CPU Resource Limit Enforcement ‚Äî ‚ùå RESOLVED (tab removed)

**Status:** The `SettingsResources` tab has been removed entirely.

**Rationale:** The `cpuLimitPercent` slider saved to store but had zero runtime effect ‚Äî no Rust handler reads it. CPU percentage limiting on Linux requires cgroups v2, `cpulimit`, or `systemd-run`, none of which are appropriate for this project. The `ramLimitMb` slider only applied to Java/Go via env vars, not as a general memory governor. The only meaningful resource control ‚Äî `threadPoolSize` (concurrent job semaphore) ‚Äî already lives in Settings ‚Üí App Engine where it belongs.

**What remains:** `executor.rs` uses hardcoded defaults (80% core count, 4GB RAM) for env-var enforcement on runtime install steps (`CARGO_BUILD_JOBS`, `GOMEMLIMIT`, `_JAVA_OPTIONS`, `NODE_OPTIONS`). These are not user-configurable ‚Äî they're sensible defaults for all runtime install/update/remove operations.

**Files changed:**

- Deleted `apps/desktop/src/renderer/src/pages/settings/SettingsResources.tsx`
- Removed import, nav entry, subtype keys from `SettingsShell.tsx`
- Removed `ResourcesSettingsSchema`, type, store key from `packages/shared/src/schemas.ts`
- Removed `"resources_settings"` from Rust store allowlist (`utils.rs`, `lib.rs`)
- Removed 10 i18n keys from all 3 locales (en-US, de-DE, ar-SA)
- `executor.rs` switched from store lookup to hardcoded const defaults

## 10. Security Surface

### 10.1 Profiles Page ‚Äî Container Status Detection ‚Äî ‚ùå KNOWN ISSUE

**Checked:** `ProfilesPage.tsx` ‚Äî the `checkRunning` function attempts to detect which profiles have running Docker containers by querying `docker ps` output. Multiple approaches were tried:

- `dh:host:exec` with `maintenance_docker_ps_table` command ‚Üí output truncation + case sensitivity issues
- `window.dh.dockerList()` API ‚Üí response parsing mismatch (rows vs data, name vs names)
- Dedicated backend endpoint `dh:profile:running-status` ‚Üí created but still not producing correct results in the frontend

**Impact:** The Start/Stop/Restart buttons on the Profiles page may show incorrect state (e.g., showing "Start" when containers are actually running, or "Stop" when they're not). The Dashboard correctly shows container status via its own `dockerList()` integration ‚Äî the issue is specific to the Profiles page's detection logic.

**Mitigation:** Users can verify actual container status on the Docker page or Dashboard. Profiles page buttons still function correctly ‚Äî Start will start compose, Stop will stop it ‚Äî the visual state indicator is just unreliable.

stop btn is not correctly working. eg, i started a pforile then clicked at stop btn, the btn will switch to start status, but the profile still runing !

**Fix path:** Rewrite `checkRunning` to use the same container data pipeline that the Dashboard uses (`DashboardMainPage.tsx` reads container state correctly). Or add a `dh:profile:running-status` result directly to the `dh:docker:list` response.

### 10.2 sshpass Password Visibility ‚Äî ‚úÖ FIXED

**Checked:** `host_exec.rs` lines 188‚Äì196: `exec_sshpass_ssh` uses `sshpass -e` with `SSHPASS` environment variable, **not** `-p` flag. The password is passed via `env("SSHPASS", password)` which is invisible in `/proc/<pid>/cmdline`. Single call site at `system_info.rs:1058` (remote SSH key setup).

**Prior audit incorrectly reported `-p` flag** ‚Äî this was already switched to `-e` before the audit. ‚úÖ **Fixed.**

### 10.3 Runtime Install Sudo Password ‚Äî ‚ùå STILL OPEN

**Checked:** `runtime_jobs.rs` line 24: `sudo_password: String` is a parameter of `runtime_job_execute`. This value comes from the `dh:job:start` IPC payload. For system-method runtime installs (apt/dnf/pacman), the user's sudo password travels through the Tauri IPC channel as plaintext JSON.

The prior audit conflated this with the Docker install flow. Docker install now correctly uses `pkexec` (Polkit). Runtime installs do not. ‚ùå NOT FIXED.

### 10.3 `/tmp` Predictable Path ‚Äî üÜï NEW FINDING (LOW RISK)

**Checked:** `system_info.rs` line 544: `let tmp = format!("/tmp/lumina_hosts_{}", std::process::id())`. This is a predictable temp file path ‚Äî potential TOCTOU / symlink attack if another process renames it before the `sudo cp`. On a single-user desktop this is negligible risk, but it violates secure temp file creation best practices.

**Fix:** Use `mktemp` or `tempfile` crate.

### 10.4 Prior P0: Docker Install Password in IPC Body ‚Äî ‚úÖ FIXED

Confirmed: `docker_engine.rs` contains no `password` field handling. Readiness wizard uses `pkexec` (polkit native dialog) at `readiness.rs:232`. This concern is resolved. ‚úÖ
