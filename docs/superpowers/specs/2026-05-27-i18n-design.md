# i18n Full Coverage — Design Spec

**Date:** 2026-05-27
**Status:** Approved

---

## Goal

Wire react-i18next across every UI component in LuminaDev. Three languages: `en-US` (English), `de-DE` (German), `ar-SA` (Arabic). Arabic uses RTL text direction — not full layout mirror. All static UI strings translated; dynamic data values (container names, file paths, repo URLs, port numbers, terminal output) stay untranslated.

---

## Non-Goals

- Full layout mirroring (sidebar flip, flex-row reversal) — not in scope
- Languages beyond en-US, de-DE, ar-SA
- Translating backend error codes, container names, file paths, or any runtime data
- Pluralization (not needed for current string set)
- CDN or HTTP-based translation loading

---

## Library Stack

| Package | Purpose |
|---------|---------|
| `i18next` | Core engine |
| `react-i18next` | React bindings (`useTranslation`, `Trans`) |
| `i18next-resources-to-backend` | Dynamic `import()` of JSON files — Vite module graph caches as singletons, no network requests |

No `i18next-browser-languagedetector` — language is read from Tauri store (`language_settings.locale`), not browser APIs.

**Security surface:** zero network calls at runtime, no eval, no crypto. JSON files bundled by Vite at build time. Module graph caches them as singletons — no accumulator growth, no memory leak.

## Key Naming Convention

Dot-separated, noun-first, camelCase leaf:

```text
container.start       → "Start"
container.stopAll     → "Stop All"
image.pull            → "Pull Image"
error.daemonDown      → "Docker daemon is not running"
status.running        → "Running"
status.stopped        → "Stopped"
```

No abbreviations. No snake_case. No raw English sentences as keys.

### Interpolation (template literals)

Any string with a dynamic value uses i18next interpolation — `{{variable}}` in JSON, passed as second arg to `t()`:

```json
{ "container.started": "Container {{name}} started" }
```

```tsx
t('docker:container.started', { name: container.name })
```

**Rule:** never concatenate translated strings with `+` or template literals. Every variable appearing inside a translated sentence must use `{{variable}}` syntax. This ensures word order is correct in German and Arabic (both differ from English).

### Trans component (JSX inside translations)

Use `<Trans>` only when the translated string must contain interactive JSX — links, bold text, buttons inline in prose:

```tsx
<Trans i18nKey="docker:install.docLink" components={{ a: <a href={url} /> }}>
  See the <a>installation guide</a> for details.
</Trans>
```

If the JSX is purely cosmetic (a `<span>` for color), extract the label and style outside the translation. `Trans` is for semantically meaningful inline elements only.

### Dynamic enum values (status labels)

Container status (`running`, `stopped`, `exited`, `paused`), job state, runtime health — translated at display time via `t()`, never mapped by a static object:

```tsx
// In common.json:
// { "status.running": "Running", "status.stopped": "Stopped", "status.exited": "Exited" }
t(`common:status.${container.state}`)  // resolves at render, falls back to EN
```

**Rule:** any enum rendered as a human-readable label gets a `common:status.*` or `common:state.*` key. Raw backend values (the string `"running"` itself) are never shown to users — always pass through `t()`.

## i18n.ts Init

```ts
import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'
import resourcesToBackend from 'i18next-resources-to-backend'

i18n
  .use(resourcesToBackend((lang: string, ns: string) =>
    import(`./locales/${lang}/${ns}.json`)
  ))
  .use(initReactI18next)
  .init({
    fallbackLng: 'en-US',
    defaultNS: 'common',
    ns: ['common','nav','dashboard','docker','git','ssh','runtimes',
         'monitor','maintenance','profiles','settings','readiness'],
    interpolation: { escapeValue: false },
    missingKeyHandler: import.meta.env.DEV
      ? (_lngs, ns, key) => console.error(`[i18n] Missing key: ${ns}:${key}`)
      : undefined,
  })

export default i18n
```

## I18nContext.tsx (Thin Bridge — No Listeners)

```ts
// One-shot only: read Tauri store → changeLanguage() → done.
// No addListener, no on(), no watch, no subscription.
// useEffect cleanup is empty because nothing is registered.
useEffect(() => {
  void window.dh.storeGet({ key: 'language_settings' }).then((res) => {
    const locale = res?.data?.locale ?? 'en-US'
    void i18n.changeLanguage(locale)
    document.documentElement.lang = locale.split('-')[0]
    if (locale === 'ar-SA') {
      document.documentElement.dir = 'rtl'
      document.documentElement.classList.add('hp-rtl')
    } else {
      document.documentElement.dir = 'ltr'
      document.documentElement.classList.remove('hp-rtl')
    }
  })
}, []) // runs once on mount — no dependencies, no re-subscription
```

`setLocale()` exposed for SettingsLanguages — calls `changeLanguage()` + updates DOM attributes + persists to store. Same pattern, same one-shot.

## RTL Strategy

Arabic = RTL text direction only. No sidebar flip, no flex-row reversal.

**What gets RTL:**

- Text alignment: paragraphs, labels, headings → `text-align: right` under `.hp-rtl`
- Inline flow: inline elements flow right-to-left naturally via `dir="rtl"` on `<html>`
- Form inputs, selects, textareas: text aligns right

**What stays LTR always:**

- Code blocks, `<code>`, `<pre>`, terminal output
- File paths, URLs
- Port numbers, IP addresses, version strings
- All numeric data displays

**Which attribute to use:**

| Attribute | Use when | Examples |
|-----------|----------|---------- |
| `data-ltr` | Mixed prose + technical content; full block stays LTR | file paths, terminal output, code snippets, URLs, container IDs |
| `data-numeric` | Pure numbers or units in a stats context; inline embed, surrounding text can still be RTL | CPU %, RAM MB, port numbers in a table cell |

Rule: if in doubt, use `data-ltr`. It is safe to over-apply. `data-numeric` only when the number sits inline inside Arabic prose and you want the surrounding text to remain RTL.

**Implementation:** single `theme/rtl.css` file, imported once in `main.tsx`:

```css
/* Text alignment */
.hp-rtl p,
.hp-rtl label,
.hp-rtl h1, .hp-rtl h2, .hp-rtl h3, .hp-rtl h4 {
  text-align: right;
}

/* Inputs align right */
.hp-rtl input:not([type="number"]),
.hp-rtl textarea,
.hp-rtl select {
  text-align: right;
}

/* Code, terminal, paths — always LTR */
.hp-rtl code,
.hp-rtl pre,
.hp-rtl .mono,
.hp-rtl [data-ltr] {
  direction: ltr;
  unicode-bidi: plaintext;
  text-align: left;
}

/* Numbers stay LTR inline */
.hp-rtl [data-numeric] {
  direction: ltr;
  unicode-bidi: embed;
}
```

## ar-SA Rollout Gate

Arabic is gated behind `beta_features_state.rtl_arabic` flag. `SettingsLanguages.tsx` shows ar-SA option only when the flag is enabled. Once RTL is verified page-by-page, flag is removed and ar-SA ships normally.

de-DE has no gate — layout-safe, no RTL risk.

## Migration Phases

### Phase 1 — Install + Init + common + nav (no page regressions possible)

1. `pnpm add i18next react-i18next i18next-resources-to-backend`
2. Write `scripts/gen-i18n-stubs.js`, run `pnpm i18n:stubs` → generates all 36 `{}` stub files
3. Write `i18n.ts`
4. Rewrite `I18nContext.tsx` as thin bridge (no listeners)
5. Import `i18n.ts` in `main.tsx` before `<App />`
6. Import `rtl.css` in `main.tsx`
7. Delete `translations.ts`
8. Wire `common` + `nav` namespaces: extract strings, populate JSON (EN + DE + AR), wire `t()` in AppShell, TopBar, ActiveJobsStrip
9. Enable de-DE in `SettingsLanguages.tsx`; ar-SA behind `rtl_arabic` beta flag

App renders correctly throughout — fallback to `en-US` means no visible change until a namespace JSON is populated.

### Phase 2 — Per-file loop

**Per-file protocol (one commit per logical unit):**

1. Read the file(s), extract every hardcoded UI string
2. Add keys to namespace JSON for all 3 languages in one pass
3. Replace strings with `t('ns:key')` in the component
4. Add `data-ltr` / `data-numeric` to technical-content elements in that file
5. Switch UI to DE, visually verify no overflow or truncation
6. Commit: `feat(i18n): wire <page> translations`

**File order and commit granularity:**

1. `DockerPage.tsx` — one commit (highest string density)
2. `DeveloperGitPage.tsx` + all `gitVcs*` sub-components — **one commit** (20+ files, one namespace, batch to avoid a dozen micro-commits)
3. `GitConfigPage.tsx` — one commit
4. `CloudGitPage.tsx` + `CloudGitActivityPanel.tsx` — one commit
5. `RuntimesPage.tsx` — one commit
6. `DashboardMainPage.tsx` + `DashboardKernelsPage.tsx` + `DashboardLogsPage.tsx` — one commit
7. `MonitorPage.tsx` + `SystemPage.tsx` — one commit
8. `MaintenancePage.tsx` — one commit
9. `SshPage.tsx` — one commit
10. `ProfilesPage.tsx` — one commit
11. All 18 `Settings*.tsx` files — one commit
12. `ReadinessWizardPage.tsx` + `SystemReadinessPage.tsx` — one commit
13. `TerminalPage.tsx` — one commit

## Test Plan

**Vitest smoke test per namespace** (`i18n.smoke.test.ts`):

```ts
// Structural test: every EN key exists in DE and AR.
// Correctness of DE/AR values is NOT asserted here —
// that requires a native speaker review before shipping.
// The raw-key guard (not.toBe(key)) applies to AR only,
// where Arabic characters guarantee the value differs from the key.
for (const ns of NAMESPACES) {
  const en = await import(`./locales/en-US/${ns}.json`)
  const de = await import(`./locales/de-DE/${ns}.json`)
  const ar = await import(`./locales/ar-SA/${ns}.json`)
  for (const key of Object.keys(en)) {
    // Structure: key must exist in all locales
    expect(de[key]).toBeDefined()
    expect(ar[key]).toBeDefined()
    // Arabic values must not equal the key string (catches copy-paste EN placeholders)
    expect(ar[key]).not.toBe(key)
    // DE correctness: toBeDefined() only — native speaker review before release
  }
}
```

**DE translation policy:** DE JSON ships with correct German from the start (AI-assisted + developer review). The smoke test asserts structure only, not correctness. A native German speaker reviews DE strings before de-DE is publicly enabled.

**Runtime dev guard:** `missingKeyHandler` logs `console.error` for any key used in a component but absent from the JSON. Catches drift immediately during development.

**Fallback assertion:** `fallbackLng: 'en-US'` means a missing DE/AR key renders in English — never as a raw key string.

## Success Criteria

- Switching to de-DE: all UI labels in German, no raw key strings visible, no text overflow
- Switching to ar-SA (behind flag): all UI labels in Arabic, text flows right-to-left, code/paths/numbers remain LTR
- Switching back to en-US: no residual state, no memory leak
- `pnpm test` passes including i18n smoke tests
- `pnpm smoke` green throughout migration (one logical unit at a time, never a broken build)
- Native speaker review of DE strings completed before de-DE is enabled by default
